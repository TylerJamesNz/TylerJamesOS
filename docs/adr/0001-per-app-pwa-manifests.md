# 0001. Per-app PWA manifests, single service worker

**Status:** Accepted, 2026-05-10.

## Context

TJOS hosts multiple top-level apps from a single origin (`/finance`, `/todos`, plus the hub at `/` and the brand kit at `/brand-kit`). The user wants Finance and Todos each installable as their own iOS home-screen icon with their own name, theme colour, and start URL, despite living on the same origin.

iOS Safari's "Add to Home Screen" reads the manifest declared in the document head at install time. It respects `scope` and `start_url`, so installs from different scopes can land as separate icons if the manifests are scoped accordingly. Beyond the iOS quirk, the alternatives are: a single TJOS-wide manifest (one icon, no per-app branding), or splitting apps onto subdomains (loses session-share, doubles the deploy).

A complication: `vite-plugin-pwa` is happiest when it generates and serves a single manifest. Letting it do so leaves no slot for per-scope manifests.

## Decision

- Each top-level app has its own static manifest file under `app/public/<app>/manifest.webmanifest` with `scope` and `start_url` fixed to that app's path, and its own name, theme colour, and icon set.
- The HTML head's `<link rel="manifest">` href is swapped on route change so iOS reads the manifest matching the user's current scope at install time.
- A single service worker generated by `vite-plugin-pwa` sits at the origin root and caches across all scopes. The plugin is configured with `manifest: false` so it does not produce its own manifest and we own the manifest files directly.
- The TJOS hub at `/` is intentionally not installable on its own. Installation only fires from inside an app's scope.

## Consequences

- Finance and Todos install as separate icons. Tapping either lands in standalone mode at that app's start URL.
- iOS standalone mode does not share cookies with mobile Safari, so the first launch of each installed PWA prompts a one-off Google sign-in inside that scope. Both PWAs share the Supabase session within their own standalone surface because they share the origin.
- Adding a new installable app (a future `/notes`, say) means adding `app/public/notes/manifest.webmanifest` and an icon set. No build configuration changes needed.
- The manifest-link-swap on route change is a small piece of glue (an effect in `App.tsx`); breaking it silently breaks installability. Worth a smoke test in the Phase I commit.
- Reversing this decision later would orphan any installed icons whose `start_url` we change. Treat scope and start_url like contracts.
